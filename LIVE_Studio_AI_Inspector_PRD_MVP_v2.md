# Live Studio AI Inspector MVP PRD v2

## 一、项目背景

TikTok LIVE Studio 产品团队负责桌面端开播工具的产品设计、功能上线和体验验证。当前功能验收和体验走查主要依赖产品经理、测试同学和主播视角的人工操作，容易受个人经验、时间投入和覆盖范围影响。

本项目希望利用 AI 辅助完成 Live Studio 功能体验和上线前验收，让 AI 能够基于指定角色和任务目标，观察并操作 Live Studio，记录体验过程，最终输出结构化的问题列表和本地报告，帮助产品经理更快发现功能缺陷、体验阻塞和流程理解问题。

本期定位为内部 vibe coding 比赛 MVP，需要有独立 GUI 程序形态，但不追求完整商业化能力。MVP 的核心目标是跑通「配置任务 → 环境检测 → 调用 Codex 判断 → RPA 操作 Live Studio → 输出本地报告」闭环。

## 三、产品目标

### 3.1 MVP 目标

1. 提供一个独立 GUI 程序，满足比赛对独立产品形态的要求。
2. 支持产品经理通过自然语言输入体验或验收任务。
3. 支持选择不同主播或测试角色，角色来自本地 Markdown 文件。
4. 支持检测本机 Live Studio、Codex CLI、桌面操作能力和屏幕信息。
5. 支持调用 Codex CLI 作为 AI 大脑，对截图和上下文进行单步判断。
6. 支持通过 DPI-aware RPA 进行截图、定位、鼠标点击等真实桌面操作。
7. 支持在执行中展示当前阶段和实时日志。
8. 支持任务结束后展示问题列表，并打开本地报告和日志目录。

### 3.2 非 MVP 目标

以下能力不进入首版强依赖：

- 飞书文档自动上传。
- 全程录屏。
- 多任务队列。
- 模型自由切换。
- 角色文件 GUI 编辑器。
- 系统托盘和桌面浮层。
- 复杂权限、安全护栏和账号管理。

## 四、方案结论

本项目采用「GUI 编排 + Codex CLI 作为 AI 大脑 + RPA Helper 作为桌面操作手」的方案。

不建议把 Codex CLI 理解为桌面操作器。Codex CLI 更适合做判断、推理、报告总结和结构化输出；真正的截图、鼠标移动、点击、窗口检测需要由 GUI 内置的 RPA Helper 完成。

推荐架构：

```text
GUI / exe
  ├─ 任务配置与状态展示
  ├─ 环境检测
  ├─ 调用 Codex CLI exec
  ├─ 调用 RPA Helper
  ├─ 过程日志
  └─ 本地报告生成

Codex CLI
  ├─ 根据任务、角色、当前截图判断下一步动作
  ├─ 输出固定 JSON schema
  └─ 总结体验问题和报告内容

RPA Helper
  ├─ 截图读屏
  ├─ DPI 适配
  ├─ 目标元素定位
  ├─ 鼠标移动与点击
  └─ 页面变化检测
```

## 五、核心流程

### 5.1 执行前

用户打开 GUI 后进入执行前页面，完成任务配置，并查看环境检测结果。

主要动作：

1. 输入任务描述。
2. 选择报告类型。
3. 选择执行角色。
4. 设置最多操作步数和截图读屏间隔。
5. 勾选是否允许真实开播。
6. 勾选是否保存关键截图作为报告证据。
7. 确认环境检测均可用。
8. 点击「开始任务」。

### 5.2 执行中

任务开始后，GUI 自动进入执行中页面。

主要动作：

1. GUI 触发截图。
2. 将任务上下文、角色摘要、当前截图和历史摘要传给 Codex CLI。
3. Codex CLI 返回结构化动作建议。
4. RPA Helper 执行动作，例如点击、等待、输入或请求人工介入。
5. GUI 更新当前阶段和实时日志。
6. 如遇到验证码、登录失效、权限弹窗、账号异常等场景，在当前阶段容器中显著提示需要人工协助。
7. 用户可点击「终止任务」中断流程。

### 5.3 执行后

任务完成、失败或被终止后，GUI 进入执行后页面。

主要动作：

1. 展示任务结果摘要。
2. 展示问题列表。
3. 每个问题展示标题、描述、严重程度和截图缩略图。如该问题没有截图证据，则不展示缩略图。
4. 用户可打开本地报告。
5. 用户可打开日志目录。
6. 用户可点击「回到首页」返回执行前状态。

## 六、交互设计与字段说明

### 6.1 设计稿预览控制

设计稿顶部包含三个按钮：

| 控件 | 说明 | 是否属于真实产品窗口 |
| --- | --- | --- |
| 执行前 | 设计稿预览按钮，用于查看执行前状态 UI | 否 |
| 执行中 | 设计稿预览按钮，用于查看执行中状态 UI | 否 |
| 执行后 | 设计稿预览按钮，用于查看执行后状态 UI | 否 |

真实产品中不展示这三个按钮。真实产品根据任务生命周期自动切换状态。

### 6.2 执行前页面

执行前页面由两个纵向模块组成：任务配置、环境检测。

#### 6.2.1 任务配置模块

| 字段 / 控件 | 类型 | 默认值 | 是否必填 | 说明 |
| --- | --- | --- | --- | --- |
| 任务描述 | 多行文本输入 | 示例任务文案 | 是 | 产品经理用自然语言输入希望 AI 完成的体验或验收任务。 |
| 报告类型 | 分段选择 | 体验报告 | 是 | 可选「体验报告」或「验收报告」。体验报告强调角色视角、路径理解、操作负担和感受；验收报告强调功能是否符合预期、缺陷和阻塞问题。 |
| 执行角色 | 下拉选择 | 萌新游戏主播 | 是 | 指定 AI 代入的体验角色。角色来自本地角色 Markdown 文件。 |
| 角色目录 | 按钮 | 无 | 否 | 打开本地角色 Markdown 文件存储目录，方便用户查看或维护角色文件。 |
| 最多操作步数 | 数字输入 | 30 | 是 | 限制单次任务最多执行多少步，避免 AI 陷入循环或长时间不可控执行。建议范围 5-80，MVP 默认 30。 |
| 截图读屏间隔 | 下拉选择 | 2 秒 | 是 | AI/RPA 每隔多久截图并读屏一次，用于判断页面是否变化、是否加载完成、是否卡顿。选项为 0.5 秒、1 秒、1.5 秒、2 秒、3 秒、5 秒。 |
| 允许 AI 执行真实开播操作 | 勾选框 | 勾选 | 是 | 显式确认 AI 可以执行真实开播相关点击。未勾选时，任务不得点击最终开播确认类按钮。 |
| 保存关键截图作为报告证据 | 勾选框 | 勾选 | 否 | 保存关键步骤截图和问题截图，用于报告证据和问题复盘。 |
| 开始任务 | 主按钮 | 无 | 是 | 点击后进入执行中状态，启动环境复检、截图读屏、Codex 判断和 RPA 操作循环。 |

#### 6.2.2 环境检测模块

环境检测模块放在任务配置模块下方，不横向并列。

| 检测项 | 状态示例 | 说明 |
| --- | --- | --- |
| Live Studio | 可用 / 未安装 / 未打开 | 检测本机是否安装 Live Studio，以及进程是否运行。若未打开，可引导用户打开或由程序尝试启动。 |
| Live Studio 登录状态 | 已登录 / 需人工确认 | 检测当前是否已进入可操作状态。由于登录态可能无法稳定自动识别，异常时进入人工确认。 |
| Codex CLI | 可调用 / 未安装 / 未登录 / 调用失败 | 检测 Codex CLI 是否存在、是否可执行、是否有 ChatGPT 登录态。 |
| 桌面操作能力 | 可用 / 不可用 | 检测截图、DPI 感知、鼠标移动和点击能力是否可用。 |
| 屏幕信息 | 已记录 / 异常 | 记录主显示器分辨率、DPI 缩放等信息，用于坐标换算和截图定位。 |

飞书 CLI 不作为 MVP 环境检测项。首版输出本地 Markdown 报告即可。

### 6.3 执行中页面

执行中页面只展示任务执行所需信息，不展示问题列表和问题统计。

#### 6.3.1 运行指标

| 字段 | 说明 |
| --- | --- |
| 已执行步数 | 当前已执行步骤数 / 最大步骤数，例如 7 / 30。 |
| 运行时间 | 从点击开始任务到当前的累计耗时。 |
| 截图读屏间隔 | 当前任务采用的截图读屏间隔，例如 2s。 |

执行中不展示「已发现问题」指标，避免用户在过程中被未确认问题干扰。问题沉淀统一在执行后展示。

#### 6.3.2 执行监控模块

| 字段 / 控件 | 类型 | 说明 |
| --- | --- | --- |
| 当前阶段 | 状态容器 | 展示 AI 当前正在做什么，例如「识别页面」「执行动作」「等待页面变化」「生成报告」。 |
| 当前操作说明 | 文本 | 展示当前步骤的简短说明，例如「Step 7 · 正在点击开始直播按钮，并等待页面反馈」。 |
| 人工协助提醒 | 状态容器内高亮提示 | 仅当需要人工介入时显示，例如登录失效、验证码、权限确认、账号异常、Live Studio 弹窗无法自动处理。 |
| 实时日志 | 日志区 | 按时间输出 observe、action、done、issue、ask_human 等过程记录。 |
| 终止任务 | 主操作按钮 | 执行中用于停止任务。它与执行前的「开始任务」是同一个主按钮在不同状态下的表现。 |

#### 6.3.3 人工协助状态

默认情况下不展示独立的「问题与接管」模块。

当需要人工介入时，在「当前阶段」容器内显著提示：

```text
需要人工协助
检测到登录失效，请完成登录后点击继续检测。
```

可触发人工协助的场景：

- 登录失效。
- 验证码。
- 账号安全确认。
- 系统权限弹窗。
- 直播资质或合规确认。
- AI 连续多次无法识别下一步。
- 页面长时间无变化且无法判断原因。

### 6.4 执行后页面

执行后页面展示任务结果和问题列表。

#### 6.4.1 任务结果摘要

| 字段 / 控件 | 类型 | 说明 |
| --- | --- | --- |
| 任务状态 | 文本 | 展示「任务已完成」「任务已终止」或「任务失败」。 |
| 执行摘要 | 文本 | 展示执行步数、耗时、问题数和本地报告生成状态。 |
| 打开本地报告 | 按钮 | 打开本地 Markdown 报告文件。 |
| 打开日志目录 | 按钮 | 打开本次任务的日志、截图和调试产物目录。 |
| 回到首页 | 按钮 | 返回执行前页面，开始新的任务配置。 |

#### 6.4.2 问题列表

每个问题条目包含：

| 字段 | 说明 |
| --- | --- |
| 问题标题 | 简短描述问题，例如「开播失败无错误提示」。 |
| 问题描述 | 说明触发路径、观察到的现象和对用户的影响。 |
| 严重程度 | Critical / Major / Minor。 |
| 截图缩略图 | 如有问题截图，则在问题条目右侧展示缩略图；如无截图则不展示。 |

严重程度定义：

| 严重程度 | 判断标准 |
| --- | --- |
| Critical | 导致任务无法完成、开播失败、核心功能不可用或用户无法恢复。 |
| Major | 明显影响任务效率或理解，用户需要额外尝试、等待或绕路。 |
| Minor | 体验不佳、文案不清、术语难懂、轻微布局或反馈问题。 |

执行后页面不再提供「上传到飞书」按钮。飞书上传作为二期能力。

## 七、Codex 调用策略

### 7.1 Codex CLI exec 的角色

`codex exec` 是 Codex CLI 的一次性任务执行模式。GUI 每一步把必要上下文传给 Codex，Codex 只返回本步判断结果。

Codex 不负责真实鼠标点击。它负责判断：

- 当前页面是什么状态。
- 下一步应该点击、等待、输入还是请求人工协助。
- 当前体验是否存在问题。
- 问题是否需要记录进报告。

### 7.2 单步短上下文

每次调用 Codex 时，只传当前判断必需的信息：

- 当前任务目标。
- 当前角色摘要。
- 当前页面截图或截图摘要。
- 最近几步操作摘要。
- 当前已知问题摘要。
- 固定输出 schema。

不把完整历史、完整 PRD、全部截图一次性传入，避免 token 过高和判断不稳定。

### 7.3 压缩历史

长任务中，历史记录需要压缩成摘要，例如：

```text
已打开 Live Studio；已进入开播设置页；摄像头权限已允许；发现摄像头列表加载等待约 4.2 秒。
```

而不是每一步都传完整日志和所有截图。

### 7.4 固定 Schema

Codex 每一步返回固定 JSON，方便 GUI 解析。

示例：

```json
{
  "action": "click_button",
  "target": "Go LIVE",
  "reason": "这是进入开播流程的主按钮",
  "need_human": false,
  "issue": null
}
```

建议动作类型：

| action | 说明 |
| --- | --- |
| observe | 继续观察当前页面。 |
| click_button | 点击指定按钮或控件。 |
| paste_text | 输入或粘贴文本。 |
| wait_change | 等待页面变化。 |
| ask_human | 请求人工协助。 |
| finish | 结束任务并生成报告。 |

## 八、RPA 操作策略

RPA Helper 负责真实桌面操作。

核心能力：

1. DPI-aware 截图。
2. 主显示器和窗口坐标换算。
3. 基于像素、图像特征或 OCR 的元素定位。
4. 鼠标移动、悬停、点击。
5. 点击前截图确认。
6. 点击后等待页面变化并截图复核。
7. 保存关键截图和问题截图。

RPA 每一步建议流程：

```text
截图 → 识别目标 → 裁剪验证 → 移动鼠标 → 点击 → 等待 → 再截图 → 判断页面变化
```

## 九、报告输出

MVP 输出本地 Markdown 报告。

报告内容建议包括：

1. 任务信息：任务描述、角色、报告类型、执行时间。
2. 环境信息：Live Studio 版本、屏幕信息、Codex CLI 状态。
3. 执行摘要：执行步数、耗时、完成状态。
4. 问题列表：标题、严重程度、描述、截图路径。
5. 过程摘要：关键步骤和观察结论。
6. 原始日志目录：便于排查和复盘。

飞书文档上传作为二期能力，不影响 MVP 任务完成。

## 十、边界 Case

| 场景 | MVP 处理策略 |
| --- | --- |
| Live Studio 未安装 | 环境检测显示不可用，任务不可开始。 |
| Live Studio 未打开 | 提示用户打开，或由程序尝试启动。 |
| Live Studio 未登录 | 当前阶段提示需要人工协助。 |
| 验证码或安全验证 | 暂停任务，请求人工处理。 |
| 权限弹窗 | 优先尝试识别并提示人工确认，必要时暂停。 |
| 页面长时间不变化 | 记录等待耗时，超过阈值后判断为疑似卡顿或请求人工协助。 |
| AI 连续无法判断 | 达到重试次数后请求人工协助。 |
| 达到最多操作步数 | 自动终止并生成当前报告。 |
| 用户点击终止任务 | 停止后进入执行后页面，报告标记为用户终止。 |
| 截图失败 | 终止任务，提示桌面操作能力异常。 |
| Codex CLI 调用失败 | 终止任务，保留日志并提示检查 Codex 登录态。 |

## 十一、验收标准

MVP 通过标准：

1. GUI 可正常启动。
2. 执行前页面可完成任务配置。
3. 环境检测能显示 Live Studio、Codex CLI、桌面操作能力、屏幕信息。
4. 点击开始任务后进入执行中页面。
5. 执行中页面能展示已执行步数、运行时间、截图读屏间隔、当前阶段和实时日志。
6. 需要人工协助时，能在当前阶段容器内显著提示。
7. 执行中可终止任务。
8. 执行完成后进入执行后页面。
9. 执行后能展示问题列表。
10. 问题条目能展示标题、描述、严重程度和可选截图缩略图。
11. 可打开本地报告。
12. 可打开日志目录。
13. 可回到首页重新开始任务。

## 十四、结论

MVP 应聚焦可运行和可演示的闭环，不追求过度配置。GUI 作为独立产品承载任务配置、环境检测和过程展示；Codex CLI 负责判断和总结；RPA Helper 负责真实桌面操作。当前交互稿已经收敛为适合 MVP 的三状态设计，可以作为开发实现依据。
